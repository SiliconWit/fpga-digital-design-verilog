## The Two-Flip-Flop Synchroniser

{/* CONTRIBUTOR: the standard fix for a single-bit signal crossing domains. Stress it works for one bit,
   not for buses. */}

```verilog title="synchronizer.v"
module synchronizer (
    input  wire clk,
    input  wire async_in,
    output wire sync_out
);
    reg ff1, ff2;
    always @(posedge clk) begin
        ff1 <= async_in;
        ff2 <= ff1;
    end
    assign sync_out = ff2;
endmodule
```

:::tip[Instrument it]
Clock-domain bugs are intermittent, so reproduce them on purpose before trusting a fix. In simulation, drive the two domains with clocks at awkward, non-integer ratios and run long. On hardware, expose the FIFO full and overflow flags on spare pins and watch them on a logic analyser under load. If the design ships in the field, those same counters (overflow events, dropped samples) are exactly the kind of health metric an [edge node can publish](https://siliconwit.io) so a rare fault is caught in the log rather than in person.
:::

## Application Questions and Solutions

### Question 1: The bug that only appears in hardware

A data path passes a multi-bit counter value from a fast clock to a slow one using a two-flip-flop synchroniser on each bit. It works most of the time but occasionally reads a wildly wrong value. Why?

<details>
<summary>**Click to reveal the solution**</summary>

<Steps>

1. **Spot the misuse.** A two-flip-flop synchroniser is safe for a single bit only. Across a multi-bit bus, individual bits can settle on different clock edges. ✅

2. **Understand the result.** During a transition the captured word can mix old and new bits, giving a value that never actually existed. ✅

3. **Use the right structure.** Pass the data through a FIFO, or convert the pointer to Gray code so only one bit changes at a time. ✅

</Steps>

</details>
